## 1. プロジェクト概要

### 1.1 背景・目的

クライアントは競馬データ分析ソフトウェア「CrossFactor」を運用しており、独自の映像解析データ（レース中の不利・距離ロス・走行位置・トラックバイアス等）を含む高精度な競馬データを保有している。

このデータを活用したAIによる予測結果を、会員制Webサービスとして提供する。競馬ファン向けに「他のサービスでは得られない独自データに基づくAI予測」を付加価値として訴求し、サブスクリプション収益を得ることが目的である。

### 1.2 システムの目標

| 目標 | 内容 |
|------|------|
| AI回収率目標 | バックテストにおいて **105%以上** を目指す |
| 対象レース | JRA（中央競馬）全レース |
| サービス形態 | 会員制Webサービス（無料 / プロプラン） |
| MVP納品期限 | 2026年4月18日（土） |

### 1.3 ビジネスモデル

- 7日間の無料トライアル期間を設け、サービスを体験させたうえでプロプランへの転換を促す
- プロプランはStripe Billingによる月額サブスクリプションで課金

### 1.4 業務要件サマリー

**誰に・何を・なぜ提供するか**

| 軸 | 内容 |
|---|---|
| **誰に** | 競馬ファン（主にJRAで馬券を購入する成人ユーザー） |
| **何を** | CrossFactor独自データ（映像解析・位置取り指数等）を活用した、他サービスでは入手できないAI予測情報 |
| **どのような価値を** | 単純な予想ではなく「期待値」ベースの馬券判断をサポートし、回収率105%超を目指す根拠あるデータを提供 |
| **なぜ今** | CrossFactorのデータ資産を活用したマネタイズ手段がなく、今回のAI予測サービスがその第一歩となる |

**主なアクター**

| アクター | 役割 | 主な操作 |
|---------|------|---------|
| **管理者（Shibata様）** | サービスオーナー・CrossFactor運用者 | CrossFactorの起動・データ確認、バッチ実行状況の確認、Stripeによる売上確認 |
| **開発・運用担当（Koshi）** | システム管理者 | Supabase/Vercel/Stripe の設定管理、ユーザーサポート対応、バッチエラー対応 |
| **プロプランユーザー** | 有料会員 | レース予測閲覧・馬券検討 |
| **未登録ユーザー** | 見込み顧客 | トップページ閲覧・無料トライアル登録 |

## 3. ユーザー像・ユースケース

### 3.1 ユーザー像

| ユーザー種別 | 説明 |
|------------|------|
| **未登録ユーザー** | 出馬表の基本情報のみ閲覧可。AIの予測データは一切見えない |
| **無料トライアルユーザー** | 登録後7日間はプロプランと同等の機能を利用可能 |
| **プロプランユーザー** | 月額課金後、全予測データ（勝率・連対率・複勝率・期待値）を閲覧可 |

### 3.2 主要ユースケース

#### UC-01: ユーザー登録〜トライアル開始

```
1. ユーザーがWebサイトにアクセス（未登録状態）
2. 「無料で試す」ボタンをクリック
3. メールアドレスとパスワードを入力して登録
4. 登録完了メールを受信・確認
5. 7日間のトライアル期間が開始（プロプランと同等の機能が利用可能）
6. トライアル終了後、自動的に未登録状態へ降格
```

#### UC-02: プロプランへの加入

```
1. ユーザーが「プロプランに登録する」をクリック
2. Stripe Checkoutへリダイレクト
3. クレジットカード情報を入力して決済
4. Stripe Webhookを受信 → Supabase上のプランを即時「pro」へ更新
5. ユーザーはリロード後にプロプランの全機能を利用可能
```

#### UC-03: AI予測結果の閲覧

```
1. ユーザー（プロ）がWebサイトにアクセス
2. レース一覧（当日・前日）から対象レースを選択
3. 出馬表と合わせてAI予測データを閲覧
   - 予想勝率・予想連対率・予想複勝率・期待値
4. 単勝・複勝の期待値を参考に馬券を検討
```

#### UC-04: 解約

```
1. ユーザーが解約手続きを行う（Stripeポータルより）
2. Stripe Webhook受信（subscription.deleted）
3. Supabase上のプランを即時「free」へ降格
4. 次回アクセス時から予測データが非表示
```

#### UC-05（管理者）: バッチ実行状況の確認・手動対応

```
1. 管理者（Shibata様 / Koshi）がバッチエラーのメール通知を受信
2. Windowsのタスクスケジューラのログ、またはPythonログファイルでエラー内容を確認
3. 必要に応じてスクリプトを手動で再実行
4. Supabaseダッシュボードで最新データが正しくアップロードされていることを確認
```

> **注:** 管理専用UIはMVP対象外。Supabase / Stripe / Vercel の各管理ダッシュボードを直接利用する。  
> 将来フェーズで管理画面を実装する場合は別途要件定義を行う。

#### UC-06（管理者）: ユーザー管理・プラン手動変更

```
1. 管理者（Koshi）がSupabaseダッシュボードにアクセス
2. users テーブルを参照し、対象ユーザーのプランカラムを確認
3. 必要に応じて plan カラムを手動で更新（例: 解約対応・テストユーザー設定）
4. Stripeダッシュボードで対応する請求状態を確認
```

#### UC-07（バッチ）: データ同期〜AI推論〜Web表示

```
1. Windowsタスクスケジューラが指定時刻に起動
   - 前日夜 / 当日朝 / レース直前
2. Python抽出スクリプトがCrossFactor SQLiteからデータを抽出
3. 特徴量生成・AIモデル推論を実行（勝率・連対率・複勝率・期待値を算出）
4. 推論結果をSupabase REST APIへアップロード
5. Next.js（Vercel）がSupabaseから最新データを取得してWeb表示
6. エラー発生時はメールで通知
```

## 4. システム概要

### 4.1 システム全体アーキテクチャ

```
【Windows（クライアントPC）】
  CrossFactor（競馬ソフト）
    └─ SQLite DB（JRA-VAN由来の競馬データ + 独自映像解析データ）
         └─ Python バッチスクリプト（タスクスケジューラで定期実行）
              ├─ データ抽出・特徴量生成
              ├─ AIモデル推論（LightGBM等）
              └─ Supabase REST API へ結果をアップロード

【クラウド】
  Supabase（クラウドDB・認証基盤）
    ├─ PostgreSQL DB（出馬表・予測結果・ユーザー情報）
    ├─ Auth（メール+パスワード認証 / JWT）
    ├─ RLS（プラン別の行レベルアクセス制御）
    └─ Edge Functions（Stripe Webhook受信）

  Vercel
    └─ Next.js App（TypeScript）
         ├─ 出馬表・AI予測結果の表示
         ├─ プラン別UI出し分け
         └─ Stripe Checkoutへの誘導

【決済】
  Stripe Billing
    └─ サブスクリプション管理 → Webhook → Supabase → プラン即時反映
```

### 4.2 技術スタック

| レイヤー | 技術 | 理由 |
|---------|------|------|
| フロントエンド | Next.js（App Router）/ TypeScript | SSR/SSGとAPIルートの統合、型安全性 |
| UIスタイル | Tailwind CSS / shadcn/ui | Figmaからの実装効率化、保守性 |
| ホスティング | Vercel | Next.js特化、Preview環境、Webhook受信 |
| クラウドDB | Supabase（PostgreSQL） | Auth/RLS/Edge Functionsの統合管理 |
| 認証 | Supabase Auth（メール+パスワード） | JWT発行、RLSとの連携 |
| 決済 | Stripe Billing | サブスクリプション、Webhook |
| AI/ML | Python / LightGBM（→ XGBoost/CatBoostへ拡張予定） | 学習速度・特徴量解釈性 |
| データソース | CrossFactor SQLite | クライアント既存環境 |
| バッチ実行 | Windowsタスクスケジューラ | 既存Windows環境を活用 |

#### AI特徴量（主要候補）

**馬の能力系**
- 近5走の着順・タイム・上がりタイム
- 距離適性・コース適性
- 前走からの距離変化

**CrossFactor独自（差別化ポイント）**
- 前走の距離ロス補正タイム
- 前走の不利有無・度合い
- トラックバイアス補正値
- 走行ポジション（先行/差し/追込）

**レース条件系**
- 距離・芝/ダート・馬場状態・天候
- クラス・賞金・枠番

**騎手・調教師**
- 騎手の近走勝率・コース別成績
- 調教師の近走勝率
- 調教タイム・本数・強さ（坂路・ウッドチップ）

**オッズ系**
- 単勝オッズ（レース締め切り直前）
- 人気順

## 6. 非機能要件

### 6.1 パフォーマンス

| 項目 | 基準 |
|------|------|
| ページ初期表示（Vercel SSR） | 2秒以内（通常時） |
| Supabase APIレスポンス | 1秒以内（通常のSELECT） |
| バッチ処理（抽出〜推論〜アップロード） | 1レース当たり5分以内 |
| Stripe Webhook処理 | 受信から30秒以内にDB反映 |

### 6.2 可用性・信頼性

| 項目 | 基準 |
|------|------|
| Webフロントエンド（Vercel） | 99.9%以上（Vercel SLAに準拠） |
| クラウドDB（Supabase） | 99.9%以上（Supabase SLAに準拠） |
| バッチ処理（Windowsローカル） | ベストエフォート（エラー時はメール通知） |
| バッチエラー通知 | 失敗後15分以内にメール送信 |

### 6.3 セキュリティ

| 項目 | 方針 |
|------|------|
| 認証 | Supabase Auth + JWT（HS256） |
| 通信暗号化 | 全通信をHTTPS/TLS 1.2以上で強制 |
| APIアクセス制御 | Supabase RLS によるプラン別データ遮断 |
| Stripe Webhook検証 | Stripe署名（`stripe-signature`ヘッダ）を必ず検証 |
| 環境変数管理 | APIキー・DB接続情報は環境変数（Vercel / ローカル `.env`）で管理、コードへの直書き禁止 |
| SQLiteアクセス | CrossFactor SQLiteはローカル完結（外部公開なし） |

### 6.6 運用・監視（Monitoring）

#### 監視対象と検知手段

| 監視対象 | 検知手段 | 通知先 | 対応フロー |
|---------|---------|-------|----------|
| **Webフロントエンド（Vercel）** | UptimeRobot（無料プラン・5分間隔）でエンドポイント死活監視 | メール | Vercelダッシュボードで原因確認 → 必要に応じてロールバック |
| **Supabase（クラウドDB）** | Supabase Status Page（https://status.supabase.com）+ Supabase Alert | メール | Supabase側の障害であれば復旧を待つ。データ破損の場合はバックアップから復元 |
| **バッチ処理（Windows）** | Pythonスクリプト内のtry-exceptで捕捉 → SMTP送信 | クライアントのメールアドレス | ログファイルで原因確認 → 手動再実行 または Koshi対応 |
| **Stripe Webhook受信** | Supabase Edge Function内でログ出力 | — | Stripeダッシュボードでイベント再送 |

> **UptimeRobot設定:** 無料プランで最大50エンドポイントを5分間隔で監視可能。  
> `https://{本番ドメイン}/` と `https://{本番ドメイン}/api/health` の2点を監視する。

#### 異常発生時の通知フロー

```
【Webフロントエンド障害】
  UptimeRobot が HTTP 非200を検知
    → UptimeRobot がメール通知（Koshi宛）
    → Koshi が Vercel ダッシュボードで確認
    → 必要に応じてロールバック or Shibata様へLINEで報告

【バッチ処理失敗】
  Pythonスクリプトが例外キャッチ
    → SMTPでクライアント（Shibata様）+ Koshi宛にメール送信（15分以内）
    → 原因確認 → Koshi が手動再実行 or スクリプト修正
    → 再実行完了後、Supabase に最新データが入っていることを確認

【Supabase障害】
  Supabase Status Page で障害検知（手動確認 or アラート）
    → ユーザーへの影響: データ取得不可（ページエラー）
    → Vercel の静的エラーページを表示（"現在メンテナンス中"）
    → Supabase復旧後、バッチを手動再実行して最新データを補完

【Stripe Webhook未着】
  Stripe Dashboard の Webhookイベント履歴で未配信を確認
    → Stripe Dashboard から手動再送
    → Supabase上のユーザープランを手動更新
```

#### 自動切り離しの方針

| シナリオ | 自動切り離し | 理由 |
|---------|-----------|------|
| Vercel障害 | Vercelが自動でロールバック（デプロイ失敗時） | Vercelのビルトイン機能 |
| Supabase接続タイムアウト | Next.jsでエラーバウンダリ表示（ユーザーへの影響を最小化） | 自動復旧型エラーハンドリング |
| バッチ処理の途中失敗 | ベースデータのみ保持・予測は未更新扱い（`12_抽出スクリプト設計書.md` 参照） | データ整合性を優先 |
| WindowsPC停止 | 自動切り離しなし（手動対応） | ベストエフォート運用 |

## 12. 用語集

| 用語 | 説明 |
|------|------|
| CrossFactor | クライアントが使用する競馬データ分析ソフトウェア。JRA-VAN等のデータ＋独自映像解析データを保有 |
| JRA-VAN | 日本中央競馬会（JRA）が提供する競馬データ配信サービス |
| SQLite | CrossFactorが使用するローカルデータベースの形式 |
| Supabase | オープンソースのBaaS（Backend as a Service）。PostgreSQL・Auth・RLS・Edge Functionsを統合提供 |
| RLS（Row Level Security） | Supabaseの行レベルセキュリティ。ユーザーのプランに応じて、取得できるデータ行を自動フィルタリングする |
| JWT（JSON Web Token） | 認証情報をエンコードしたトークン。Supabase Authが発行し、Next.jsとSupabaseの通信で使用 |
| Stripe Billing | 定期課金（サブスクリプション）を管理するSaaS |
| Stripe Webhook | Stripeからサーバーへ送られるイベント通知。決済完了・解約等の際にプランを自動更新するために使用 |
| プロプラン | 月額課金後に利用できる有料会員プラン。AI予測データの全項目が閲覧可能 |
| 無料トライアル | 新規登録後7日間、プロプランと同等機能を無料で試せる期間 |
| LightGBM | Microsoft製の勾配ブースティング機械学習ライブラリ。学習速度が速く、競馬予測のベースラインモデルとして採用 |
| SHAP | 機械学習モデルの予測根拠を説明するためのライブラリ（SHapley Additive exPlanations） |
| 回収率 | （払戻金額合計 / 購入金額合計）× 100。100%以上でプラス収支 |
| 期待値 | 予想勝率 × 単勝オッズ（または予想複勝率 × 複勝オッズ）。1.0以上が「買い」の目安 |
| トラックバイアス | コースの内側・外側など特定コースポジションが有利・不利になる傾向（CrossFactor独自データ） |
| 距離ロス | 競走馬が最短距離でなく外側を走ることで生じた余剰距離（CrossFactor独自データ） |
| タスクスケジューラ | Windows標準の定期実行機能。指定した時刻に自動でバッチスクリプトを起動する |
| MVP（Minimum Viable Product） | 最小限の機能で市場に投入できる製品。今回は3週間で構築するシステムの第1版を指す |
| Vercel | Next.jsに特化したホスティングサービス。Gitと連携し、プッシュするだけで自動デプロイ |
