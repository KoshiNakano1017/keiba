# AI実行基盤比較

**最終更新:** 2026-03-30  
**対象:** 競馬AIの推論・期待値計算・バックテスト・再学習の実行基盤  
**前提技術:** Python / SQLite / LightGBM / XGBoost / CatBoost / Supabase

---

## 1. この資料の目的

本資料は、今回のプロジェクトにおける **AI実行基盤** を比較するための整理資料です。

ここでいう AI実行基盤には、以下を含みます。

- CrossFactor由来データの前処理
- AI推論
- 期待値の計算
- バックテスト
- 必要に応じた再学習

一方で、以下は本資料の対象外です。

- Next.js Webアプリの配信
- ログイン画面や会員ページの表示
- Stripe課金導線

これらは `05_ホスティング比較.md` で扱います。

---

## 2. 前提条件

今回の案件では、AI実行基盤の選定にあたり、以下の制約が重要です。

- 元データは `CrossFactor` の `SQLite` にあり、クライアントのローカルWindowsが起点
- クライアントは海外出張が多く、ローカルPCを常時触れるとは限らない
- 初期フェーズでは、`LightGBM` などCPU中心のベースライン運用が現実的
- Web表示よりも、`データ鮮度`、`安定実行`、`再実行のしやすさ` が重要

---

## 3. 比較対象（MECE整理）

| カテゴリ | 候補 | 概要 |
|---------|------|------|
| ローカル実行型 | クライアントWindows | CrossFactor DBに最も近い既存PC上で処理する |
| 汎用PaaS型 | Render Worker / Cron | Pythonジョブを比較的簡単にクラウドで回せる |
| サーバレスジョブ型 | Cloud Run Jobs / Service | 必要時だけバッチや推論を実行しやすい |
| 仮想マシン型 | 小さめVM | 常時起動・長時間処理に向く自由度の高い構成 |
| 高性能計算型 | GPU付きクラウド | 将来の大規模学習や高速化向け |

---

## 4. 結論サマリー

### 推奨順位

1. **ローカルWindowsで抽出 + Cloud Run JobsでAI処理**
2. **ローカルWindowsで抽出 + 小さめVMでAI処理**
3. **ローカルWindows常時起動 + 遠隔操作**
4. **Render Worker / Cron**
5. **GPU付きクラウド**

### 結論

- **最有力はハイブリッド構成**
  - `CrossFactorがローカルWindows起点` という前提を活かしつつ、AI処理の安定性はクラウド側で確保できます。
- **ローカルWindows単独運用は、MVPでは成立しても本番運用ではリスクが残る**
  - 特に、海外出張中の停止・再起動・障害対応に弱いです。
- **Renderは簡単だが、AI実行基盤としてはCloud RunやVMより中途半端になりやすい**
  - 常時起動ワーカーには向く一方、長時間バッチや費用最適化では不利です。

## 5. コストと重要項目の一覧比較

> 注記: 以下の金額は、AI実行基盤としての概算です。  
> `Supabase`、`Webホスティング`、`Stripe手数料` は別途発生します。

| 候補 | 導入コスト | ランニングコスト | コスト感の評価 | その他重要項目 |
|------|------------|------------------|----------------|----------------|
| クライアントWindows | 低 | 低 | ○ | 既存資産を使えるが、可用性と遠隔運用が弱い |
| Render Worker / Cron | 低〜中 | 中 | △ | 分かりやすいが、ジョブ追加や常時起動で積み上がりやすい |
| Cloud Run Jobs / Service | 中 | 低〜中 | ◎ | 従量課金で無駄が少なく、ジョブ実行と相性がよい |
| 小さめVM | 中 | 中 | ○ | 長時間処理に強く自由度も高いが、OS保守が必要 |
| GPU付きクラウド | 高 | 高 | △ | 初期MVPには過剰、深層学習や高速学習向け |

### 5-1. コスト感の目安

| 候補 | 導入コストの目安 | ランニングコストの目安 | 備考 |
|------|------------------|------------------------|------|
| クライアントWindows | 既存PCならほぼ追加なし | 電気代中心、月 `1,000〜4,000円` 程度目安 | 24時間稼働やUPS導入で別途費用が出る |
| Render Worker / Cron | 数時間〜1日程度 | 月 `3,000〜15,000円+` 程度 | WorkerやJobが増えるほど積み上がりやすい |
| Cloud Run Jobs / Service | 1〜2日程度の設計を見込みやすい | 月 `数百円〜8,000円+` 程度 | 実行回数、メモリ、ログ、通信量で変動 |
| 小さめVM | 1〜2日程度 | 月 `2,000〜10,000円+` 程度 | 常時起動やスペック増で上振れする |
| GPU付きクラウド | 数日以上を見込みやすい | 月 `数万円〜` もありうる | GPU利用時は費用が跳ねやすい |

### 5-2. AI実行基盤の選定で特に重要な項目

今回の案件では、以下の項目を特に重視すべきです。

- `CrossFactor SQLite` に依存するため、ローカルWindowsとの橋渡しが必要
- `長時間処理` や `再実行` に耐えられること
- `ログ保存` と `障害時の切り分け` がしやすいこと
- クライアント不在時でも、できるだけ運用が止まらないこと
- 将来的な `再学習` や `条件違いのバックテスト` に対応しやすいこと

## 6. 候補別の比較

### 6-1. クライアントWindows

**カテゴリ:** ローカル実行型

| 観点 | メリット | デメリット | 評価 |
|------|----------|------------|------|
| コスト | 既存PCを使えるなら初期費用が小さい | 専用機化やUPS導入で追加費用が出る | ○ |
| 保守しやすさ | CrossFactor DBに最も近く、抽出が簡単 | 停止時の復旧が人手依存になりやすい | △ |
| 改修しやすさ | Python中心なら実装しやすい | 個人PC依存になりやすく、再現性が落ちやすい | △ |
| セキュリティ | データをローカルで閉じやすい | 遠隔操作や外部公開の設計を誤ると危険 | △ |
| 速度 | ローカルDB直読みで速い | PCスペックと他作業の影響を受ける | ○ |
| 負荷への強さ | 小規模なら成立する | 長時間処理や連続ジョブに弱い | △ |
| その他 | 立ち上がりが最も早い | 海外出張中の停止・再起動問題が残る | △ |

## 7. ローカルWindows運用案の比較

クライアントのローカルWindowsを使う場合、運用案は以下のように整理できます。

| 案 | 概要 | メリット | デメリット | 評価 |
|----|------|----------|------------|------|
| 1 | 必要に応じて遠隔起動 | 普段は電源OFFで省電力 | Wake on LANやネットワーク条件に依存し、失敗時に詰まりやすい | △ |
| 2 | PCをつけっぱなしにして遠隔操作 | 最も分かりやすく、すぐ始めやすい | 電気代、Windows Update、停電、フリーズ対策が必要 | ○ |
| 3 | ノートPCにシステムを入れる | 可搬性がある | 発熱、スリープ、電源断に弱く、本番運用には不向き | △ |
| 4 | 出張先のデスクトップにも環境を入れる | ローカルで作業できる | 環境差分と同期事故のリスクが高く、運用が崩れやすい | × |

### 補足

- 4案の中では `2. PCをつけっぱなしにして遠隔操作` が最も現実的です
- ただし、最適解は `ローカルWindowsは抽出専用、AI処理はクラウド` へ役割分担することです

## 8. 推奨方針

### 推奨案A（第一候補）

- ローカルWindowsでは `CrossFactor SQLite` からの抽出だけ行う
- 抽出結果をクラウドへ送る
- `Cloud Run Jobs` で推論・期待値計算・バックテストを実行する
- 結果は `Supabase` に保存する

#### この案のメリット

- ローカル依存を最小化できる
- クライアント不在時でもAI処理が回しやすい
- コストと拡張性のバランスがよい
- 将来の再学習やジョブ追加にも対応しやすい

#### この案のデメリット

- 初期設計は少し増える
- ローカルとクラウドの橋渡し設計が必要

---

### 推奨案B（次点）

- ローカルWindowsでは抽出のみ行う
- `小さめVM` で推論・バックテスト・再学習を実行する

#### この案のメリット

- 長時間処理に強い
- Python環境の自由度が高い
- Cloud Runより直感的で分かりやすい

#### この案のデメリット

- OS保守と監視が必要
- 常時起動の固定費が出る

---

### 非推奨案

- クライアントの通常利用PCをAI実行基盤の本番中核にする
- 複数のローカルPCへ同じ環境を配布して二重運用する

#### 理由

- 海外出張中の停止や復旧に弱い
- 環境差分で障害原因が分かりにくくなる
- 属人化しやすく、保守性が落ちる

---

## 9. クライアント向けの一言要約

今回のAIは、`CrossFactorのローカルWindows` だけで完結させるよりも、`ローカルではデータ抽出、クラウドではAI処理` と役割分担した方が、安定して運用しやすくなります。  
そのため、AI実行基盤としては `Cloud Run Jobs` を第一候補、`小さめVM` を第二候補とし、ローカルWindows単独運用は補助的な位置づけで考えるのが妥当です。
