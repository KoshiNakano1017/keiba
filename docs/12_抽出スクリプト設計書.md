# 抽出スクリプト設計書

**最終更新:** 2026-03-30  
**対象:** CrossFactor SQLite -> 特徴量生成用データセット -> Supabase  
**実装言語:** Python  
**実行環境:** Windows PC / タスクスケジューラ

---

## 1. 目的

本設計書は、`CrossFactor` の `SQLite` から競馬データを抽出し、AI推論およびWeb表示に必要な形式へ整形して `Supabase` へ連携するための抽出スクリプトの設計方針を定義する。

本スクリプトは単なるCSV書き出しではなく、以下を担う。

- SQLiteから必要テーブルを安全に読み出す
- レース単位・出走馬単位のキーを正規化する
- AI特徴量に必要な列を抽出する
- 締め切り直前オッズを選別する
- 異常データ・欠損データを検知する
- 推論スクリプトおよびSupabase投入処理へ受け渡す

---

## 2. スコープ

### 2.1 本スクリプトが担当する範囲

- `CrossFactor SQLite` への接続
- 対象テーブルの抽出
- レースキー・馬キーの生成
- 差分対象レースの選定
- 特徴量元データの整形
- 推論入力用データセットの出力
- Web表示用のベースデータ出力
- Supabase投入用ペイロード生成
- ログ出力
- エラー通知トリガー

### 2.2 本スクリプトが担当しない範囲

- LightGBMモデルの学習
- Webフロントの表示ロジック
- Stripe / 認証 / 課金処理
- Supabaseの最終テーブル設計そのもの

> ただし、将来の `Supabase` テーブル設計を見据えた出力形式は本設計書で定義する。

## 4. 対象テーブル

MVPで抽出対象とする主要テーブルは以下とする。

| 区分 | テーブル | 用途 |
|------|---------|------|
| 開催・レース | `JV開催スケジュール` | 対象開催日の特定 |
| 開催・レース | `JVレース詳細EX` | レース条件・開催情報 |
| 出走馬 | `JV馬毎レース情報EX` | 出走馬ごとの主要特徴量 |
| マスタ | `JV競走馬マスタ` | 馬齢・性別・血統登録番号など |
| マスタ | `JV騎手マスタ` | 騎手名・所属など |
| マスタ | `JV調教師マスタ` | 調教師名・所属など |
| オッズ | `JVオッズ_単複枠` | 単勝・複勝オッズ、人気順算出 |
| 払戻 | `JV払戻` | バックテスト正解データ |
| 調教 | `JV坂路調教` | 調教生データの補完確認用 |
| 調教 | `JVウッドチップ調教` | 調教生データの補完確認用 |

### 4.1 Phase 1 の主軸テーブル

Phase 1 の主軸テーブルは以下で確定とする。

- `JVレース詳細EX`
- `JV馬毎レース情報EX`
- `JVオッズ_単複枠`
- `JV払戻`

この4テーブルで、MVPに必要な以下をほぼ満たせる。

- レース条件
- 出走馬ごとの主要特徴量
- 締め切り直前オッズ
- バックテスト用の払戻

### 4.2 補助テーブル

主軸ではないが、MVPでも補助的に利用する候補は以下。

- `JV競走馬マスタ`
- `JV騎手マスタ`
- `JV調教師マスタ`
- `JV開催スケジュール`

### 4.3 Phase 1 で主軸にしない候補

以下は Phase 1 で無理に主軸にしなくてよい。

- `JV馬毎レース情報EX2`
- `JVレース詳細EX2`
- `JVオッズ_馬連`
- `JVオッズ_3連複`
- `JVオッズ_3連単`

これらを主軸にすると、MVPの要件である `単勝・複勝` 中心の予測に対して設計が重くなりやすい。

### 4.4 優先的に使う考え方

`JV馬毎レース情報EX` には `CrossFactor` がすでに算出済みの派生指標が多数格納されているため、MVPでは以下の方針を採る。

- まず `JV馬毎レース情報EX` の計算済み列を最優先で利用する
- 生テーブルからの再計算は最小限に抑える
- Phase 2以降で必要に応じて独自再計算を追加する

## 5. 抽出単位

### 5.1 主キー

レース単位の主キーを以下で統一する。

```text
race_key =
  開催年 + 開催月日 + 競馬場 + 開催回 + 開催日目 + 番号
```

出走馬単位の主キーを以下で統一する。

```text
entry_key =
  race_key + 馬番
```

馬個体の主キーは以下とする。

```text
horse_key =
  血統登録番号
```

### 5.2 抽出対象期間

用途別に期間を分ける。

| 用途 | 抽出対象 |
|------|---------|
| 当日表示用 | 前日夜〜当日レース直前の対象開催日 |
| 推論用 | 当日出走分 + 必要な過去参照データ |
| 学習/バックテスト用 | 過去 `5〜10年` |

## 9. 抽出ロジック設計

### 9.1 レース抽出

1. `JV開催スケジュール` から対象日を取得
2. `JVレース詳細EX` から対象日の `JRA` レースを取得
3. `地方競馬` 相当レコードは除外
4. `race_key` を付与

抽出したい主な項目:

- `開催年月日`
- `競馬場`
- `開催回`
- `開催日目`
- `番号`
- `距離`
- `競走種別`
- `天候`
- `芝馬場状態`
- `ダ馬場状態`
- `出走頭数`
- `発走時刻`

### 9.2 出走馬抽出

1. `JV馬毎レース情報EX` から対象 `race_key` の出走馬を取得
2. `異常区分` や除外情報を確認
3. `entry_key` を付与
4. Phase 1で必要な特徴量元列を抽出

優先抽出列:

- `血統登録番号`
- `馬番`
- `枠番`
- `斤量`
- `馬体重`
- `馬体重増減`
- `平均着順_前3走`
- `平均着順_前5走`
- `平均人気_前3走`
- `着度数P_馬場適性`
- `着度数P_競馬場`
- `着度数P_同距離`
- `位置取り指数`
- `予想脚質`
- `4角5番手内率`
- `上がり3位内率`
- `PCI`
- `休養週数`
- `距離増減`
- `斤量補正値`
- `競走馬単勝回収率`
- `競走馬複勝回収率`
- `坂路_直近1_4F合計`
- `坂路_直近1_1F`
- `_直近1_4F合計`
- `_直近1_1F`

### 9.3 マスタ結合

対象:

- `JV競走馬マスタ`
- `JV騎手マスタ`
- `JV調教師マスタ`

用途:

- 馬齢算出
- 性別取得
- 騎手名・調教師名の表示用補完
- 将来の血統特徴量対応の土台作成

### 9.4 オッズ抽出

1. `JVオッズ_単複枠` から対象 `race_key` の全時点オッズを取得
2. `発表月日時分` の最大値を締め切り直前候補として選択
3. 取得時点の妥当性をチェック
4. 単勝・複勝オッズを特徴量へ連携
5. 選択した時点が `結果確定後` でないことを検証する
6. 当日朝バッチでは暫定値、レース直前バッチでは最大 `発表月日時分` を最終値として採用する

> バックテストでは必ず「結果確定前の最終時点オッズ」を使う。

### 9.5 払戻抽出

`JV払戻` は学習・評価用で使用する。

用途:

- 単勝回収率計算
- 複勝回収率計算
- 正解ラベル評価

## 10. 差分更新設計

### 10.1 方針

毎回全件を再投入せず、以下を差分対象とする。

- 対象日が今回のバッチ対象に含まれるレース
- オッズ時点が更新されたレース
- 出走取消・除外など変更が入ったレース
- 再実行フラグが立ったレース

### 10.2 チェックポイント

ローカルにチェックポイントファイルを持つ。

候補:

- `last_run_at`
- `last_target_date`
- `last_race_count`
- `last_odds_timestamp_by_race`

### 10.3 Upsertキー

| データ | Upsertキー |
|-------|-----------|
| レース | `race_key` |
| 出走馬 | `entry_key` |
| 予測結果 | `entry_key + odds_snapshot_at` または `entry_key + batch_id` |

> 最終的なUpsertキーは `Supabase` テーブル設計時に確定する。

## 11. バリデーション設計

### 11.1 件数チェック

- `JVレース詳細EX` の件数 > 0
- レースごとの `JV馬毎レース情報EX` 件数 > 0
- `出走頭数` と抽出件数の大きな乖離がない

### 11.2 主キー重複チェック

- `race_key` の重複がないこと
- `entry_key` の重複がないこと

### 11.3 必須列チェック

必須列が欠損した場合は、即時失敗ではなくルールを分ける。

| 列種別 | 例 | 方針 |
|--------|----|------|
| ハード必須 | `race_key`, `entry_key`, `血統登録番号`, `馬番` | 1件でも欠損で失敗 |
| 重要特徴量 | `単勝オッズ`, `平均着順_前3走` | 欠損率しきい値超過で失敗 |
| 補助特徴量 | `坂路_直近2_4F合計` | 欠損許容、ログ出力のみ |

### 11.4 型チェック

- 数値列が文字列混入していないか
- 日付列が `YYYYMMDD` 形式か
- オッズ列のパースが可能か

## 12. エラー処理設計

### 12.1 エラー分類

| 種別 | 例 | 対応 |
|------|----|------|
| 接続エラー | SQLiteファイル未存在、ロック中 | リトライ後に失敗通知 |
| データエラー | 必須列欠損、件数異常 | 処理停止、通知 |
| 変換エラー | 型変換失敗、オッズパース失敗 | 対象行を隔離し閾値超過で停止 |
| 連携エラー | Supabase Upsert失敗 | リトライ、失敗時通知 |
| 推論エラー | モデルファイル未読込、推論失敗 | ベースデータは保持し、予測結果のみ未更新として扱う |

### 12.2 リトライ方針

| 対象 | 回数 | 間隔 |
|------|------|------|
| SQLite接続 | 3回 | 10秒 |
| Supabase API | 3回 | 5秒, 15秒, 30秒 |
| メール送信 | 2回 | 30秒 |

### 12.3 通知内容

メール本文には以下を含める。

- 実行日時
- 実行モード
- 対象日
- 失敗ステップ
- 例外メッセージ
- ログファイルの保存先
- 再実行要否

## 14. 実行モード設計

### 14.1 モード一覧

| モード | 用途 | 主な対象 |
|-------|------|---------|
| `daily_evening` | 前日夜バッチ | 翌日対象レースの初回抽出 |
| `race_day_morning` | 当日朝バッチ | 当日朝時点の更新反映 |
| `pre_race` | レース直前バッチ | 締め切り直前オッズ反映 + 推論 |
| `backfill` | 過去データ再構築 | 学習・バックテスト用 |
| `retry_failed` | 障害再実行 | 失敗した対象のみ |

### 14.2 CLI例

```bash
python run_pipeline.py --mode daily_evening --target-date 2026-03-30
python run_pipeline.py --mode pre_race --target-date 2026-03-30
python run_pipeline.py --mode backfill --from 2018-01-01 --to 2025-12-31
```

- `JVオッズ_単複枠` の `発表月日時分` 最大値を締め切り直前オッズとして扱う
- 抽出処理と推論処理は `別プロセス` で分離する
- 通知先はクライアントのメールアドレス

## 18. 現時点の推奨方針

未確定項目があるため、現時点では以下を推奨する。

- Phase 1では `JVレース詳細EX` `JV馬毎レース情報EX` `JVオッズ_単複枠` `JV払戻` を主軸にして最短で動かす
- `SQLite -> 中間Parquet -> Supabaseベース投入 -> 別プロセス推論 -> 予測投入` の5段階に分けて障害切り分けしやすくする
- 差分更新は `対象日 + race_key` 単位で行う
- オッズは `発表月日時分` 最大値を締め切り直前値として採用する
- 推論失敗時はベースデータを保持し、予測結果のみ未更新として通知する
- エラー時はメール通知する
- 再実行性を重視して `Upsert` 前提で実装する
