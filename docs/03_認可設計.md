## 1. プラン構成

### 1.1 バックエンド上の状態

| 状態 | `users.plan` | 説明 |
|------|-------------|------|
| 未登録 | — | ログインしていない状態 |
| トライアル中 | `trial` | 登録後7日間。プロ相当の閲覧権限を持つ |
| トライアル期限切れ | `expired` | トライアル終了後の制限状態 |
| プロ | `pro` | 課金済みユーザー |

### 1.2 UI設計上の表示状態

| UI表示状態 | 対応するバックエンド状態 | 用途 |
|-----------|-------------------------|------|
| 未ログイン | 未登録 | 認証前の画面表示 |
| トライアル | `trial` | 無料体験中の画面表示 |
| トライアル期限切れ | `expired` | 制限状態の画面表示 |
| プロ | `pro` | 全機能閲覧可能状態 |

> **注:** システム上のSSOTは `trial / expired / pro` であり、画面設計でも `未ログイン / トライアル / トライアル期限切れ / プロ` を基本状態として扱う。  
> `ライトプラン` は存在しない前提とする。

## 2. プラン別閲覧制限（機能マトリクス）

| 機能・データ項目 | 未登録 | トライアル中 | トライアル期限切れ | プロ |
|------------------|:------:|:------------:|:------------------:|:----:|
| レース一覧 | ○ | ○ | ○ | ○ |
| 出馬表（馬名・枠番・騎手・オッズ等） | ○ | ○ | ○ | ○ |
| 予想勝率 | モザイク | ○ | モザイク | ○ |
| 予想連対率 | モザイク | ○ | モザイク | ○ |
| 予想複勝率 | モザイク | ○ | モザイク | ○ |
| 期待値 | モザイク | ○ | モザイク | ○ |
| 課金誘導CTA | 登録誘導 | 残日数通知 | プロ登録誘導 | 非表示 |

> **凡例:** `○` = 閲覧可、`モザイク` = UI上は存在を示すが具体値は非表示

### 2.1 段階ロック設計の画面前提

| UI表示状態 | ロック表現 |
|-----------|-----------|
| 未ログイン | `★` と `▲` は表示、`⚠` と `◆` はマスクして認証誘導を出す想定 |
| トライアル | `★ / ▲ / ⚠ / ◆` をすべて閲覧可。残日数通知バーを表示する |
| トライアル期限切れ | `★` と `▲` のみ表示し、`⚠` と `◆` はマスクしてプロ登録CTAを出す想定 |
| プロ | `★ / ▲ / ⚠ / ◆` をすべて閲覧可 |

## 3. 技術実装方針

### 認証
| 項目 | 内容 | ステータス |

| 認証方式 | メール＋パスワード | **確定** |
| セッション管理 | Supabase JWT | **確定** |

### 認可（アクセス制御）
| 項目 | 内容 | ステータス |
|------|------|----------|
| 行レベルセキュリティ（RLS） | Supabase RLS でプラン別データアクセス制御 | **確定（方針）** |

| フロントエンド権限チェック | セッション + `users.plan` を参照してNext.jsでUI分岐 | **確定（方針）** |
| UI制御 | ロック表示、CTA、モーダル、遷移先制御を画面状態に応じて分岐 | **確定（方針）** |

### Stripe連携
| 項目 | 内容 | ステータス |
|------|------|----------|
| 決済基盤 | Stripe Billing（サブスクリプション） | **確定** |
| プラン変更時の即時反映 | Stripe Webhook → Supabase ユーザーテーブル更新 → RLS自動適用 | **確定（方針）** |

## 4. 認可フロー（シーケンス概略）

```
ユーザー → Stripeで決済
    ↓
Stripe Webhook → バックエンド（Next.js API Route or Supabase Edge Function）
    ↓
Supabase users テーブルの plan カラムを更新
    ↓
Supabase RLS がリクエスト時に plan を参照して自動フィルタリング
    ↓
フロントエンド（Next.js）がJWTのclaimsを参照してUI出し分け
```

## 5. 画面遷移時の認可ルール

| 画面 | トリガー | 認可ルール |
|------|---------|-----------|
| レース一覧 | レースカード押下 | 全ユーザー遷移可。詳細画面で表示制御を行う |
| レース一覧 | `ログイン / 無料登録` | 未ログイン時のみ認証画面へ遷移 |
| レース一覧 | `プロ登録` | 未ログイン時は認証誘導、ログイン済み制限ユーザーはプラン選択画面へ遷移 |
| レース詳細 | マスク要素押下 | 状態に応じてログイン誘導またはプロ登録モーダルを表示 |
| ログイン / 新規登録 | 認証完了 | `returnTo` で遷移元へ戻す |
| プラン選択 | 決済完了 | 権限再取得後に元画面へ戻し、ロック解除状態を反映する |

## 6. 画面内イベントと権限反映

| イベント | 実装上の扱い |
|---------|-------------|
| アプリ起動時の状態判定 | セッション有無と `users.plan` を取得し、UI表示状態へマッピングする |
| ヘッダー表示切替 | `未ログイン / トライアル / トライアル期限切れ / プロ` に応じてCTAを切り替える |
| マスク領域クリック | 権限が足りない場合、直接値は返さず、モーダルまたはCTAのみ表示する |
| 決済完了後の再表示 | セッション再取得または再フェッチを行い、ロック状態を即時更新する |

## 7. データモデル（Supabase側・仮）

```sql
-- users テーブル（Supabase Authと連携）
create table users (
  id uuid references auth.users primary key,
  plan text not null default 'trial', -- 'trial' | 'expired' | 'pro'
  stripe_customer_id text,
  stripe_subscription_id text,
  trial_ends_at timestamptz,
  plan_expires_at timestamptz,
  created_at timestamptz default now()
);

-- RLSポリシー（例: proのみ期待値カラム取得可）
-- ※ 詳細はデータ・マッピング表確定後に設計
```

> テーブル名・カラム名は打ち合わせ後に確定
