## 4. 認可フロー（シーケンス概略）

```
ユーザー → Stripeで決済
    ↓
Stripe Webhook → バックエンド（Next.js API Route or Supabase Edge Function）
    ↓
Supabase users テーブルの plan カラムを更新
    ↓
Supabase RLS がリクエスト時に plan を参照して自動フィルタリング
    ↓
フロントエンド（Next.js）がJWTのclaimsを参照してUI出し分け
```

## 5. データモデル（Supabase側・仮）

```sql
-- users テーブル（Supabase Authと連携）
create table users (
  id uuid references auth.users primary key,
  plan text not null default 'free', -- 'free' | 'pro'
  stripe_customer_id text,
  stripe_subscription_id text,
  plan_expires_at timestamptz,
  created_at timestamptz default now()
);

-- RLSポリシー（例: proのみ期待値カラム取得可）
-- ※ 詳細はデータ・マッピング表確定後に設計
```

> テーブル名・カラム名は打ち合わせ後に確定
