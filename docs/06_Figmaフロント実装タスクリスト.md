# Figmaからフロント実装を行うためのタスクリスト

**対象:** CrossFactor AI Webフロント  
**目的:** Figmaを起点に、保守・改修しやすい `Next.js` フロントエンドを実装するための具体的な作業手順を定義する  
**前提:** クライアントから共有されたURLが `Figma Make` の場合、実装仕様の取得元としては不十分なことがあるため、必要に応じて通常の `Figma Design file` を追加で取得する

## 方針

- `Figma Make` の見た目だけを頼りに実装を始めない
- 実装前に、画面仕様を `構造化された情報` として取得する
- `Next.js App Router` と `Tailwind CSS` を前提に、再利用しやすいUI部品へ分解する
- ページ単位の見た目再現より、`状態管理`、`アクセシビリティ`、`改修容易性` を優先する
- Figma上の値をそのままコードへ埋め込まず、必要に応じて `デザイントークン` に変換する

## 5. デザイントークン設計

- [ ] 色を `primary`, `surface`, `text`, `muted`, `border` などの役割名で整理する
- [ ] タイポグラフィを `heading`, `body`, `caption` などに整理する
- [ ] 余白を `spacing scale` として整理する
- [ ] `radius`, `shadow`, `border` を再利用可能な単位に整理する
- [ ] 頻出するFigma値は `Tailwind` のテーマ変数に寄せる
- [ ] 一時的な `arbitrary value` を増やしすぎない

### トークン設計の意図

- [ ] Figma変更時に影響箇所を追いやすくする
- [ ] UIの横展開をしやすくする
- [ ] 微妙に違う値が乱立するのを防ぐ
- [ ] デザインの一貫性を保つ

---

## 6. コンポーネント分割設計

- [ ] ページ全体をセクション単位で分割する
- [ ] 共通で使う `Button`, `Card`, `Chip`, `Badge` を先に切り出す
- [ ] `Header` の内部要素を必要に応じて分割する
- [ ] `Hero` はテキスト領域と画像領域を分離する
- [ ] `PremiumLockedSection` はロック状態の表現責務を閉じ込める
- [ ] `RaceCardList` と `RaceCardItem` を分ける
- [ ] props には意味のある名前を付ける
- [ ] Figmaのレイヤー名をそのままコード名に使わない

### コンポーネント設計ルール

- [ ] `見た目の差分` は `variant` などで表現する
- [ ] `業務データ` と `スタイル制御` のpropsを分ける
- [ ] 親が持つべき状態を子へ押し込みすぎない
- [ ] 再利用しないものは無理に共通化しすぎない

## 13. 画面設計ブループリント（Figma / React実装用）

以下は、別途整理した `3状態（未ログイン / トライアル / プロ）`、`段階ロック設計`、`レース一覧の3段構造` を前提にした画面設計の叩き台である。  
今後の `Figma` 移行、および `React` 実装時の情報設計・画面分解・状態整理の基準として利用する。

### 13.1 画面名一覧

| 画面名 | 英語名 | 役割 |
|------|------|------|
| トップ画面 / レース一覧画面 | Race List Screen | アプリの入口。開催中・今後のレースを `1レース1カードの3段構造` で一覧表示するメイン画面 |
| レース詳細・AI予想画面 | Race Detail Screen | 選択したレースのAI予想（`★ / ▲ / ⚠ / ◆`）と出馬表を表示する中核画面 |
| ログイン / 新規登録画面 | Auth Screen | 未ログインユーザー向けの認証画面 |
| プラン選択 / アップグレード画面 | Pricing Screen | `トライアル終了後の再登録` や `プロ登録` を案内する課金導線画面 |
| マイページ / アカウント設定画面 | My Page / Settings | ユーザー情報、現在プラン、契約変更導線を提供する画面 |

### 13.2 画面遷移の条件

#### A. トップ画面 / レース一覧画面からの遷移

| トリガー | 遷移先 / 動作 | 条件 |
|---------|--------------|------|
| 任意のレースカードをクリック | レース詳細・AI予想画面へ遷移 | 全ユーザー共通 |
| ヘッダーの `ログイン / 無料登録` をクリック | ログイン / 新規登録画面へ遷移 | 未ログイン時のみ |
| ヘッダーの `プロ登録` をクリック | プラン選択 / アップグレード画面へ遷移 | 未ログイン / トライアル期限切れ時を主対象 |
| ユーザーアイコンをクリック | マイページ / アカウント設定画面へ遷移 | ログイン時のみ |

#### B. レース詳細・AI予想画面からの遷移

| トリガー | 遷移先 / 動作 | 条件 |
|---------|--------------|------|
| `戻る` ボタンをクリック | レース一覧画面へ遷移 | 全ユーザー共通 |
| マスクされた予想（`⚠` / `◆`）をクリック | CTAモーダルを表示し、必要に応じて認証またはプラン選択へ誘導 | 未ログイン / トライアル期限切れ時 |
| 詳細CTAをクリック | プラン選択画面またはログイン画面へ遷移 | モーダル表示後 |

#### C. ログイン / 新規登録画面からの遷移

| トリガー | 遷移先 / 動作 | 条件 |
|---------|--------------|------|
| ログイン完了 | 直前にいた画面へリダイレクト | `returnTo` 等で遷移元を保持する |
| 新規登録完了 | 直前にいた画面へリダイレクト | レース一覧またはレース詳細へ戻す |

#### D. プラン選択 / アップグレード画面からの遷移

| トリガー | 遷移先 / 動作 | 条件 |
|---------|--------------|------|
| 決済完了 | 決済完了サンクスページ、または元のレース詳細画面へ遷移 | 実装方式に応じて選択 |
| 権限更新完了 | 対象画面のマスクを解除 | `プロ` 反映後にアンロック |

### 13.3 主要イベント・状態変化

#### A. 全体共通のイベント

| イベント | 内容 |
|---------|------|
| 権限状態の判定 | アプリ起動時にユーザー状態（`未ログイン / トライアル / プロ`）を判定し、表示UIとCTAを切り替える |
| 共通ヘッダー制御 | 権限状態に応じて `ログイン / 無料登録`、`プロ登録`、`ユーザーアイコン` の表示を出し分ける |
| リダイレクト制御 | 認証や決済後に元画面へ戻すため、遷移元コンテキストを保持する |

#### B. トップ画面 / レース一覧画面のイベント

| イベント | 内容 |
|---------|------|
| タブ切り替え | `本日` `明日` `過去のレース` などの開催日タブ押下で対象レース一覧を再描画する |
| 絞り込み | 会場（東京、中山など）やレース条件の変更に応じて一覧を即時更新する |
| カード選択 | 選択レースを詳細画面へ渡し、対象レースの表示コンテキストを引き継ぐ |

#### C. レース詳細・AI予想画面のイベント

| 状態 | 表示 / イベント |
|------|----------------|
| 未ログイン | `★` 本命と `▲` 高期待値は表示。`⚠` 危険人気馬と `◆` 穴馬はマスク表示し、クリック時に `ログイン / 登録が必要` のモーダルまたはトーストを表示する |
| トライアル | `★ / ▲ / ⚠ / ◆` をすべて表示する。無料期間と残日数表示を優先し、課金導線は通知バー中心に扱う |
| プロ | `★ / ▲ / ⚠ / ◆` をすべて表示する。ロック解除後は詳細見解をアコーディオン等で展開できる |

| 個別イベント | 内容 |
|-------------|------|
| マスク要素クリック | 状態に応じてログイン誘導またはプロ登録誘導を行う |
| 馬名 / アイコンクリック | 過去成績やAI評価詳細を表示するドロワーを開く |
| 詳細見解の開閉 | プロプラン時のみAI見解テキストをアコーディオンで開閉する |

### 13.4 実装メモ

- `Race List Screen` は `一覧表示`, `フィルタ`, `日付タブ`, `レースカード` に分割して設計する
- `Race Detail Screen` は `レース基本情報`, `予想記号セクション`, `ロック制御`, `馬詳細ドロワー` に責務分割する
- `未ログイン / トライアル / プロ` の各状態差分は、Figma上で `Variant` または `注釈` として明示する
- マスク要素のクリック時挙動は、`直接遷移` ではなく `モーダル経由` を第一候補として検討する
- React実装時は、`表示状態` と `契約状態` を分離し、コンポーネント側に課金ロジックを過剰に持たせない

- ファイル名がそのまま画面名であるため、追加共有時も同じ命名方針で統一する
- 同一画面で状態差分がある場合は、画面名に `未ログイン` `トライアル` `プロ` などを追記すると識別しやすい
- 料金プラン画面の旧 `ライトプラン` 表示は画面素材上の旧案として扱い、正本要件では採用しない
- `S06` は共有画像上の差分から仮置きしたIDであり、`未ログイン` なのか `トライアル期限切れ` なのかは追加確認後に確定する
