## 1. MVPの定義（スコープ）

### MVP範囲内（確定）
- CrossFactor（SQLite）→ Supabase へのデータ自動同期
- AIモデルによる予想勝率の算出
- Next.js製Webサイトでの予測結果表示
- Stripe決済によるサブスクリプション管理
- プラン別のデータアクセス制御（RLS）

### MVP範囲外（後回し・確定）
- ユーザー向けの詳細UI（Figma完全再現は第2フェーズ）
- 管理画面（Shibata様用のデータ操作UI）
- モバイルアプリ
- AI精度の最終チューニング（初期はベースラインモデルで可）

### MVP範囲（未定）
- 地方競馬データの対応 | 未定（Shibata様確認後）
- 無料プランの表示範囲の詳細 | 未定（権限マトリクス確定後）

## 4. 「支払い対象」の定義

> 以下を「支払い対象（検収完了）」とする

1. 上記「第3週末の最終検収条件」の必須項目がすべて合格であること
2. 実際の サーバーがわ の Windows 環境で一気通貫動作が確認できること
3. バックテスト結果が「未定の目標ライン」に準ずる合理的な説明ができること

> **注意:** AI精度の「絶対値」を検収条件にすると、初回は達成困難なため、  
> 「動くシステム＋説明可能なモデル＋バックテスト結果の提示」を検収条件とすることを推奨する。

### 6.9 テスト実行順序

テスト項目は以下の順序で実行する。前段の失敗は後段に波及するため、上から順に確認する。

1. **型チェック / lint** — `npm run type-check && npm run lint`
2. **画面スモーク確認** — W-01, W-02, W-16, W-19, W-22, W-28（画面が表示されるかの最低限確認）
3. **認証フロー確認** — W-03〜W-07（新規登録・ログイン・returnTo）
4. **権限 UI 差分確認** — W-08〜W-15, W-20〜W-21, W-23〜W-24（4 状態の表示差分）
5. **決済導線確認** — W-17〜W-18（Stripe Checkout 開始）
6. **Vercel 結合確認** — W-25〜W-27（Preview / Production 起動）
7. **エラー表示確認** — W-29（Error Boundary 発動）

### 6.10 エラー解析・修復フロー

テスト中にエラーが発生した場合、以下のフローで切り分けと修復を行う。

```
1. 再現条件を固定する
   - URL、操作手順、ユーザー状態（guest/trial/expired/pro）を記録

2. 原因層を分類する
   ┌─────────────────────┐
   │ UI 層               │ → コンポーネント描画、CSS、状態管理
   │ middleware 層        │ → セッション refresh、リダイレクト
   │ server action 層    │ → フォーム送信、認証処理
   │ repository / adapter│ → データ取得、mock ↔ Supabase 差異
   │ env / config 層     │ → 環境変数、Vercel 設定
   └─────────────────────┘

3. 一次情報を確認する
   - ブラウザ console（クライアントエラー）
   - Vercel Functions ログ（サーバーエラー）
   - Supabase Auth レスポンス（認証エラー）
   - Next.js ビルドログ（型エラー、import 解決失敗）

4. 原因を修正する
   - 修正は最小スコープにとどめ、関連箇所への影響を確認

5. 同系統ケースを再検証する
   - 修正したエラーと同じ層・同じ機能に属するテスト項目を再実行
```

### 6.11 修復結果の記録テンプレート

| 項目 | 内容 |
|------|------|
| テスト項目 | W-XX |
| 再現条件 | （URL・操作手順・ユーザー状態） |
| エラー内容 | （メッセージ・スタックトレース要約） |
| 原因層 | UI / middleware / server action / repository / env |
| 修正内容 | （変更ファイル・変更概要） |
| 再検証結果 | Pass / Fail |
| 日付 | YYYY-MM-DD |
