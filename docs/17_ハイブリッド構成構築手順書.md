## 2. GCP vs AWS：バックアップ基盤の選定結論

### 2.1 比較サマリー

| 観点 | GCP（推奨） | AWS |
|------|:---:|:---:|
| **VM月額** | **0円**（e2-micro Always Free） | $5〜10/月（Lightsail） |
| **ストレージ月額** | **0円**（GCS 5GB Free） | Lightsail内蔵 or S3（有料） |
| **無料期間** | **永久** | 3ヶ月のみ無料 |
| **従量課金リスク** | ほぼなし | なし（Lightsailは固定） |
| **リージョン** | 無料枠は米国のみ | 東京リージョンあり |
| **セットアップ難易度** | 中 | 低 |
| **ドキュメント充実度** | ◎ | ◎ |

### 2.2 結論：GCPを推奨

**理由:**

1. **コスト最優先:** ハイブリッド構成のバックアップVMは「普段使わない待機系」であり、月額0円であることが最大の価値
2. **従量課金の完全排除:** DさんがMTGで示した従量課金への懸念を100%解消できる
3. **レイテンシの許容:** 米国リージョンでもSupabaseへのアップロードは数秒の遅延で済む。障害時の一時利用なら十分許容範囲
4. **十分なスペック:** e2-micro（2vCPU / 1GB RAM）+ GCS 5GBで、LightGBMの推論バッチは実行可能
5. **永久無料:** AWSの3ヶ月無料とは異なり、GCPのAlways Freeは期限なし

### 2.3 GCP無料枠の構成

| リソース | スペック | 無料条件 |
|---------|---------|---------|
| **VM（GCE e2-micro）** | 2 vCPU / 1GB RAM | `us-west1` or `us-central1` or `us-east1` |
| **ストレージ** | 30GB Standard HDD | 上記リージョンのみ |
| **GCS（Cloud Storage）** | 5GB/月 | `us-west1` or `us-central1` or `us-east1` |
| **ネットワーク** | 1GB/月の外向き通信 | 北米から他リージョンへの送信 |

### 2.4 AWSが向くケース（参考）

以下の場合はAWS Lightsailが有利になります。

- 東京リージョンのレイテンシが必須（レース直前の数秒が致命的な場合）
- 1GB RAMでは推論処理が不足する場合（→ Lightsail $10プラン 1GB RAM）
- GCPのセットアップが難しいと判断した場合

> 現時点では上記いずれも該当しないため、GCPを第一候補とします。

## 3. 構築の全体像とフェーズ分け

### 3.1 構築フロー全体図

```
Phase 1-A  ローカルPC環境の構築（1〜2日）
  │  Python環境・バッチスクリプト・タスクスケジューラ
  │
  ├─ Phase 1-B  SaaS層の構築（1日）  ← 並行作業可
  │    Supabase / Vercel / Stripe の初期セットアップ
  │
  ├─ Phase 1-C  遠隔操作環境の構築（0.5日）  ← 並行作業可
  │    SSH or リモートデスクトップの設定・接続テスト
  │
  ▼
Phase 1-D  GCPバックアップ環境の構築（1日）
  │  GCPアカウント・VM作成・Python環境構築
  │
  ▼
Phase 1-E  バックアップ同期の構築（0.5〜1日）
  │  ローカル→GCSの自動同期・GitHubとの連携
  │
  ▼
Phase 1-F  障害時フェイルオーバー手順の整備（0.5日）
  │  手順書作成・切替テスト実施
  │
  ▼
★ 構築完了（合計 4〜6日）
```

### 3.2 フェーズ別の優先度と依存関係

| フェーズ | 内容 | 工数目安 | 前提 | MVP必須 |
|---------|------|---------|------|:------:|
| **1-A** | ローカルPC環境 | 1〜2日 | なし | ✅ |
| **1-B** | SaaS層 | 1日 | なし（1-Aと並行可） | ✅ |
| **1-C** | 遠隔操作 | 0.5日 | DさんのPC＋ネット環境 | ✅ |
| **1-D** | GCPバックアップVM | 1日 | Googleアカウント | 🔶 MVP後でも可 |
| **1-E** | バックアップ同期 | 0.5〜1日 | 1-A + 1-D完了後 | 🔶 MVP後でも可 |
| **1-F** | フェイルオーバー手順 | 0.5日 | 1-D + 1-E完了後 | 🔶 MVP後でも可 |

> **段階的アプローチ:** Phase 1-A〜1-Cを先に完了させればMVPとして稼働可能。  
> Phase 1-D〜1-Fは「保険」として後から追加できる。

## 6. Phase 1-C：遠隔操作環境の構築

### 6.1 選択肢

| 方式 | メリット | デメリット | 推奨度 |
|------|----------|------------|:------:|
| **SSH（OpenSSH）** | 軽量・セキュア・コマンドライン操作に最適 | GUIが使えない | ◎ |
| **リモートデスクトップ（RDP）** | GUI操作可能・直感的 | 帯域を消費。セキュリティ設定が必要 | ○ |
| **Tailscale（VPN）** | ポート開放不要・設定が簡単・安全 | サービスへの依存 | ◎ |
| **Chrome Remote Desktop** | 設定が最も簡単 | Googleアカウント依存。遅延が大きい | △ |

### 6.2 推奨：Tailscale + SSH の併用

| 手順 | 作業内容 |
|------|---------|
| 1 | DさんのPCにTailscaleをインストール |
| 2 | 中野晃志のPCにTailscaleをインストール |
| 3 | 同一Tailscaleネットワークに参加 |
| 4 | DさんのPCでOpenSSH Serverを有効化 |
| 5 | Tailscale内部IPでSSH接続テスト |
| 6 | 必要時はRDP（Tailscale経由）でGUI操作 |

**Tailscaleのメリット:**
- ルーターのポート開放が不要（NAT越え対応）
- 通信は暗号化済み（WireGuardベース）
- 固定IPが不要（Tailscale内部IPを使用）
- 無料プラン（最大100デバイス）で十分

### 6.3 接続テストの確認項目

| # | 確認項目 | 合格基準 |
|---|---------|---------|
| 1 | SSH接続の成功 | Tailscale IP経由でログインできる |
| 2 | コマンド実行 | `python --version` が正しく返る |
| 3 | ファイル転送 | `scp` でファイルの送受信ができる |
| 4 | バッチ実行 | SSH経由でバッチスクリプトを実行・完了確認できる |
| 5 | 再接続 | 一度切断後に再接続できる |
| 6 | RDP接続（任意） | GUI操作が必要な場合に画面を見られる |

### 7.3 GCE VM作成時の設定チェックリスト

**無料枠の条件を外すと課金が発生します。** 以下を厳守してください。

| 設定項目 | 設定値 | 理由 |
|---------|--------|------|
| マシンタイプ | **e2-micro** | これ以外は有料 |
| リージョン | **us-central1** / us-west1 / us-east1 のいずれか | 他リージョンは有料 |
| ブートディスクの種類 | **Standard persistent disk** | SSD/Balancedは有料 |
| ブートディスクのサイズ | **30GB以下** | 30GBまで無料 |
| Network Service Tier | **Standard** | Premiumは有料 |
| Ops Agent | **インストールしない** | RAM消費を避ける |
| バックアップ | **なし** | スナップショットは有料 |
| 外部IP | Ephemeral（一時的）で可 | 静的IPは未使用時に有料 |

## 8. Phase 1-E：バックアップ同期の構築

### 8.1 同期対象と方式

| 対象 | 同期先 | 方式 | タイミング |
|------|--------|------|----------|
| ソースコード | GitHub Private | `git push` | コード変更時 |
| 学習済みモデル | GitHub LFS or GCS | `git push` or `gsutil cp` | 再学習時 |
| 抽出済みデータ | GCS | バッチ終了後に自動アップロード | バッチ実行ごと（1日3回） |
| 設定ファイル | GitHub Private | `git push` | 設定変更時 |
| ログファイル | GCS（任意） | 日次バッチで古いログを転送 | 1日1回 |

### 8.2 バッチ処理への組み込み

バッチパイプライン（`run_pipeline.py`）の最終ステップにバックアップ処理を追加します。

```python
# run_pipeline.py の最終ステップ（イメージ）

def backup_to_gcs(run_dir: str, bucket: str):
    """バッチ実行結果をGCSにバックアップ"""
    import subprocess
    
    target = f"gs://{bucket}/runs/{os.path.basename(run_dir)}/"
    subprocess.run([
        "gsutil", "-m", "cp", "-r",
        os.path.join(run_dir, "*.parquet"),
        target
    ], check=True)

def backup_model_to_gcs(model_path: str, bucket: str):
    """学習済みモデルをGCSにバックアップ"""
    import subprocess
    
    target = f"gs://{bucket}/models/{os.path.basename(model_path)}"
    subprocess.run([
        "gsutil", "cp", model_path, target
    ], check=True)
```

## 9. Phase 1-F：障害時フェイルオーバー手順の整備

### 9.1 障害シナリオと対応方針

| # | シナリオ | 影響 | 対応方針 |
|---|---------|------|---------|
| 1 | DさんのPCがフリーズ | バッチが停止。当日の予測が未配信 | PC再起動 → バッチ手動再実行 |
| 2 | DさんのPCが故障 | バッチが長期停止 | GCP VMでフェイルオーバー |
| 3 | 停電 | バッチが停止 | 復電後にPC起動 → バッチ手動再実行 |
| 4 | ネットワーク障害 | Supabaseへのアップロード失敗 | リトライ → 復旧後に手動再実行 |
| 5 | Supabase障害 | データ投入・Web表示が停止 | Supabase復旧を待つ |
| 6 | Vercel障害 | Web表示が停止 | Vercel復旧を待つ（自動ロールバック） |

### 9.2 フェイルオーバー手順（PC故障時）

```
【Step 1: 検知】（所要時間: 〜数分）
  - バッチ失敗のエラー通知メールを受信
  - または、当日のレース予測がWebに反映されていないことを確認

【Step 2: 判断】（所要時間: 〜5分）
  - DさんにLINEで状況を確認
  - PCの復旧見込みを確認
  - 復旧に数時間以上かかる場合 → Step 3へ

【Step 3: GCP VMへの切替】（所要時間: 10〜20分）
  1. GCP VMにSSH接続
     $ gcloud compute ssh crossfactor-backup --zone=us-central1-a

  2. 最新コードを取得
     $ cd ~/keiba-ai && git pull origin main

  3. GCSから最新の抽出済みデータを取得
     $ gsutil -m cp -r gs://crossfactor-backup-data/runs/latest/ ~/keiba-ai/runs/

  4. 環境変数を確認
     $ cat .env  # Supabase URL/Key が正しいことを確認

  5. 推論バッチを実行
     $ source .venv/bin/activate
     $ python batch/run_pipeline.py --mode pre_race --use-cached-data

  6. Supabaseで結果を確認
     - Supabase Dashboard → predictions テーブルに最新データがあることを確認

  7. Vercel側のWeb表示を確認
     - 本番URLにアクセスして予測データが表示されることを確認

【Step 4: 継続運用】
  - PCが復旧するまでGCP VM上でバッチを手動実行
  - 復旧後はローカルPCでの運用に戻す

【Step 5: 復旧後の確認】
  - ローカルPCのバッチが正常に動作することを確認
  - GCP VMのバッチスケジュールを停止（手動実行していた場合）
```

### 9.3 フェイルオーバーテストの実施

構築完了後に以下のテストを実施し、切替手順が実際に機能することを確認します。

| # | テスト項目 | 合格基準 |
|---|----------|---------|
| 1 | GCP VMにSSH接続できる | 接続成功 |
| 2 | GitHubから最新コードを取得できる | `git pull` 成功 |
| 3 | GCSから抽出済みデータを取得できる | `gsutil cp` 成功 |
| 4 | GCP VM上でバッチを実行できる | 推論完了・エラーなし |
| 5 | Supabaseに結果がアップロードされる | predictions テーブルに新規データ |
| 6 | Vercelで予測データが表示される | Web上で確認 |
| 7 | 切替開始から完了まで30分以内 | 時間計測 |
